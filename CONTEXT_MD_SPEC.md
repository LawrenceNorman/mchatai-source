# CONTEXT.md — Artifact Continuation Spec

> Every artifact generated by AIWizard ships with a `CONTEXT.md`.
> This file lets any LLM (Claude Code, Gemini CLI, OpenAI Codex CLI) pick up
> the project cold and continue building it without the original session context.

## Why This Exists

When AIWizard generates a mini-app, pipeline, or microservice, it has full context
about what it was building and why. That context evaporates after the session ends.
`CONTEXT.md` preserves it in a machine-readable + human-readable format so that:

- Future LLM sessions can resume intelligently
- Humans can understand what was built and why
- The mchatai-source registry stays navigable as it grows
- Any LLM CLI can parse the frontmatter to route tasks correctly

---

## Schema (YAML Frontmatter + Markdown Body)

```yaml
---
artifact:
  type: miniapp | microservice | pipeline | agent | macos-app | skill | game
  id: string          # reverse-domain: official.name-slug or user.name-slug
  name: string        # Human-readable name
  version: string     # SemVer (1.0.0)
  created_by: string  # "AIWizard" or username
  created_at: string  # ISO 8601 date (YYYY-MM-DD)
  updated_at: string  # ISO 8601 date

purpose:
  summary: string           # 1-sentence: what it does
  problem_solved: string    # Why this exists
  intended_users: string    # Who uses it

architecture:
  language: string          # python | javascript | swift | html+js | typescript
  frameworks: [string]      # FastAPI, SwiftUI, React, etc.
  platform: string          # web | macos | ios | cloud | shell | cross-platform
  has_ui: boolean
  requires_auth: boolean

dependencies:
  external_apis: [string]       # APIs called (name only, no keys)
  internal_services: [string]   # Other mchatai-source services used
  storage: string               # localStorage | Firestore | SQLite | none

development:
  main_file: string             # Entry point: main.py, index.html, agent.json
  config_file: string           # manifest.json, pipeline.json, etc.
  entry_points: [string]        # Key functions/endpoints to modify first
  build_command: string         # "pip install -r requirements.txt" or "none"
  test_command: string          # How to verify it works
  deploy_command: string        # How to ship it

deployment:
  platform: string              # Cloud Run | Firebase Hosting | mChatAI | none
  environment_vars: [string]    # Required env vars (names only, never values)
  health_check: string          # Endpoint or command to verify deployed health

known_issues: [string]          # Outstanding bugs or TODOs
tags: [string]                  # Semantic tags for registry search
---
```

## Body Sections (Markdown)

```markdown
## What This Does
[1-2 sentences. Be specific about the output, not just the category.]

## How to Continue Building This
[The most common next requests and where to start.]

### Critical Entry Points
- **`FunctionName()`** in `file.ext` (line N): [What it does, when to touch it]
- **`/endpoint`** in `file.ext`: [What it handles]

### Key Concepts
[3-5 bullets of essential knowledge needed to modify this safely.
Include "don't do X because Y" gotchas.]

## Testing
[Exact steps to verify changes work, including edge cases.]

## Deployment
[Exact commands or steps to ship a change.]

## If You're Stuck
[Specific troubleshooting: symptom → cause → fix.]
```

## Size Guidelines

| Artifact Complexity | Target Length |
|---------------------|--------------|
| Simple (mini-app, single-file) | 40–60 lines |
| Medium (microservice, pipeline) | 60–100 lines |
| Complex (multi-file app, agent) | 100–150 lines |
| Large system | Split into `CONTEXT.md` (summary) + `ARCHITECTURE.md` (detail) |

---

## Scaling Path

Artifacts grow naturally through stages:

```
Stage 1 — Tiny (1-2 files):
  index.html + manifest.json + CONTEXT.md

Stage 2 — Small (3-10 files):
  + ARCHITECTURE.md  (decisions log: why choices were made)

Stage 3 — Medium (10-50 files):
  + tests/ directory
  + docs/ subdirectory for API reference

Stage 4 — Large (50+ files):
  + Worth adding to CodeGraphContext index (cgc index .)
  + CONTEXT.md becomes high-level; ARCHITECTURE.md has detail
```

`CONTEXT.md` is never replaced — it grows alongside the artifact.

## Type-Specific Extensions

Add these sections/fields for specific artifact types:

**For AI-heavy artifacts (agents, pipelines, skills):**
```yaml
llm_config:
  model: claude-opus-4-6 | gemini-2.5-pro | gpt-4o
  temperature: 0.7
  max_tokens: 8000
  prompt_file: prompt.md   # if prompt is externalized
```

**For microservices:**
```yaml
monitoring:
  health_endpoint: /health
  metrics_port: 8081
  error_rate_threshold: 0.05
```

**For games:**
```yaml
game:
  engine: html+js | unity | godot
  multiplayer: boolean
  leaderboard: boolean
  estimated_session_minutes: 5
```

## Registry Integration

When AIWizard publishes an artifact to `mchatai-source/`, it:

1. Generates `CONTEXT.md` alongside the artifact
2. Adds an entry to the registry `INDEX.md` at the repo root
3. The entry format:

```markdown
| [Name](path/) | type | tags | one-line summary |
```

This makes the entire registry navigable by both agents and humans.
