## Why no changes were needed

These references were designed specifically for **game-UI surfaces** (mini-apps with a table layout, action buttons, CPU opponents). The frontend-skill rubric's most restrictive rules (`fs-001` one-composition, `fs-002` hero budget, `fs-005` narrative flow) target **landing pages**. They do not apply to a product-UI like a card game, where the product IS the layout.

The rules that DO transfer (fs-003 cards-as-default, fs-007 copy discipline, fs-011 premium-without-decoration, fs-012 product-UI priority) are all already satisfied by the existing references.

**The real gap the audit surfaces is not a broken reference — it is a *missing* reference.** Today `layout-references/` has zero webSPA archetypes (landing pages, product UIs for a web app). When the wizard generates a webSPA and `frontend-skill-guidance` is the only frontend context in its system prompt, the LLM has no concrete example to follow. Per the v6 commentary on `aiwizard-miniapp.json`: *"rules describe, references demonstrate."*

## Added in this PR

- `webspa-product-landing.{dom.html, styles.css, motion.js, md}` — React-style HTML skeleton demonstrating hero → narrative → CTA with 3 motion snippets inlined.
- `webspa-single-screen-app.{dom.html, styles.css, md}` — product-UI pattern (orientation / status / action per fs-012), not a landing page.
- Both registered in `index.json` with keywords (landing / saas / product page / app ui / single-screen).

Together these close the webSPA gap without touching the card-game references.
